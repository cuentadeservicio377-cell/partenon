Hermes

Recibe intenciones del emprendedor, pregunta el tipo de empresa y convierte la respuesta en misiones publicadas en el Partenón.

Eval Loop

Juez de calidad que corre antes de enviar, en runtime y en producción. Cada output se scored contra una rúbrica antes de llegar al usuario.

Flujo

De la intención a la entrega.

El emprendedor no le da instrucciones a cada héroe. Habla con Hermes. Hermes publica la misión. El héroe correcto la toma y trabaja con el emprendedor hasta entregar.

Stack del hackathon

NVIDIA y Stripe como infraestructura de agentes.

Partenon no solo consume APIs. Corre dentro de un sandbox, elige modelos por tarea y ejecuta pagos con límites explícitos.

security

NVIDIA NemoClaw + OpenShell

Hermes harness

NemoClaw orquesta agentes autónomos de larga duración. OpenShell aplica políticas fuera del proceso del agente: filesystem, red, procesos y privacidad. Incluso un prompt injection no puede saltarse las restricciones.

  • check Sandbox por héroe con políticas YAML.
  • check Privacy router: datos sensibles a modelos locales.
  • check Hermes es uno de los dos harness soportados oficialmente.
memory

Nemotron 3 Ultra

550B parámetros totales, 55B activos por token, arquitectura híbrida Mamba-Transformer MoE, contexto de 1M tokens. Post-entrenado para planificación, tool calling, delegación y recuperación de errores.

  • check Hasta 5× más throughput que modelos open comparables.
  • check 30% más barato en tareas agentic.
  • check Usado por el Guardián y el Estratega para reasoning complejo.
payments

Stripe Skills

stripe-link-cli

Compras seguras con tarjeta virtual de un solo uso o Shared Payment Token. Cada spend request requiere aprobación en la app de Link.

mpp-agent

Paga por llamadas a APIs que responden HTTP 402. Soporta Stripe Link, Tempo Wallet, Privy Agent CLI y AgentCash.

stripe-projects

Provisiona SaaS autónomamente: Neon, Twilio, Vercel, Prisma, Runloop y 44 proveedores más. Credenciales sincronizadas a .env.

Safety limits

Máximo $5,000 por spend request; $5,000 diarios; $20,000 mensuales. Ventana de aprobación de 10 minutos. Tarjeta/SPT válida 12 horas. Stripe Projects aisla presupuestos por proveedor y ambiente (dev/staging/prod).

Ejemplo: provisionar Neon con Stripe Projects 
# Onboard del proyecto
stripe projects init

# Ver catálogo
stripe projects catalog

# Provisionar Postgres para la app de pedidos
stripe projects add neon/postgres --env dev

# Crear spend request para el primer mes
link-cli spend-request create \
  --merchant-name "Neon" \
  --amount 2900 \
  --context "PostgreSQL para proyecto Aurora" \
  --request-approval

# Aprobación en la app de Link; tarjeta virtual de un solo uso
# Credenciales se escriben en .env; Guardián rota keys al renovar

Runtime

Hermes es más que un modelo.

Partenon aprovecha mecanismos nativos de Hermes Agent: skills system, memoria persistente, cron, subagentes, messaging gateway, MCP y un closed learning loop.

folder_copy

Skills System

Cada héroe es un SKILL.md con frontmatter, slash commands y dependencias. Hermes puede crear o parchear skills tras misiones complejas.

storage

Memoria

MEMORY.md y USER.md inyectados en el system prompt; SQLite FTS5 para búsqueda de sesiones; G-Brain como MCP server para memoria compartida.

schedule

Cron

Tareas programadas en lenguaje natural o expresión cron. El gateway entrega a Telegram, WhatsApp, email, Slack y otras plataformas.

Subagentes

delegate_task spawnea agentes hijos aislados con toolsets restringidos. El Estratega usa subagentes para misiones que cruzan varios héroes.

forum

Messaging Gateway

Una sola sesión persiste en 22 plataformas: Telegram, WhatsApp, Slack, Discord, Signal, Email, Teams, LINE y otras.

Background review propone cambios a memoria y skills. El Curator consolida la biblioteca cada 7 días. Todo bajo aprobación humana.

Quality Layer

Slop se atrapa antes de salir.

Partenon no genera y espera. Cada output pasa por un eval loop con casos de prueba, una rúbrica y un threshold. Eso es lo que diferencia un agente productivo de un generador de slop.

flowchart LR subgraph EVAL["Eval Loop"] TC["Test Cases
ground truth"] R["Rúbrica 0-1"] TH["Threshold 0.7"] J["Judge Skill"] end H["Héroe"] -->|genera| OUT["Output"] OUT --> EVAL J -->|score >= 0.7| SHIP["Ship"] J -->|score < 0.7| REWORK["Rework + nuevo test case"] REWORK --> TC
01

Antes de enviar

Cambio de prompt o modelo dispara la suite completa. Hermes compara scores contra baseline y pide aprobación si algo regresa.

02

En runtime

Cada output se valida contra estructura y rúbrica. Si un JSON está roto o un reporte no cuadra, el héroe lo corrige antes de entregar.

03

En producción

Un cron samplea ejecuciones reales. Si el score promedio baja, Hermes avisa. Cada fallo se convierte en un nuevo caso de prueba.

MCP

G-Brain vía MCP.

MCP es el cable entre los héroes y el cerebro. Cada agente puede leer contexto, escribir resultados y consultar lo que otros aprendieron.

01 / Perfiles

Archivos .finance, .design y .security definen comportamiento.

02 / Misiones

Cada tarea queda registrada con input, output y aprendizajes.

03 / Entidades

Clientes, proveedores y productos se guardan como objetos reutilizables.

API Reference

Contratos entre héroes y G-Brain.

G-Brain expone operaciones MCP que todos los héroes usan para leer perfiles, escribir misiones y consultar entidades.

Método Args Retorno Descripción
gbrain.read_profile hero: string Perfil YAML Lee el perfil operativo de un héroe.
gbrain.write_mission hero, input, output, score mission_id Registra una misión ejecutada con eval score.
gbrain.read_missions hero?, limit? Lista de misiones Historial filtrable por héroe.
gbrain.put_entity type, id, payload entity_id Guarda cliente, proveedor o producto.
gbrain.get_entity type, id Entidad Recupera una entidad por tipo e ID.
gbrain.search_memory query, hero? Hits relevantes Búsqueda semántica sobre memoria.
Ejemplo: leer perfil 
await mcp.call('gbrain.read_profile', { hero: 'tesorero' });
// { role, permissions, pegaso, eval_rubric, ... }
Ejemplo: escribir misión 
await mcp.call('gbrain.write_mission', {
  hero: 'cobrador',
  input: { amount: 2900, merchant: 'Neon' },
  output: { spend_request_id: 'sr_...' },
  score: 0.92
});
Workshop

Paquete de implementación de 90 minutos.

El mismo formato sirve para universidades, organizaciones empresariales, coworkings y aceleradoras. Los participantes se llevan Hermes instalado.

Fase 1

Pre-instalación

antes del evento

El participante corre curl ... | bash antes de llegar. El instalador crea el entorno Python, expone hermes como comando global y copia los seis perfiles.

Python 3.11+ hermes CLI 6 perfiles
Fase 2

Setup y Google Sheet

0–20 min

Clonar el Google Sheet base, conectar variables de entorno y verificar que los héroes puedan escribir en Workspace.

Google Sheets Service account .env
Fase 3

Onboarding del tipo de empresa

20–60 min

Hermes pregunta qué empresa es. El participante responde y el router genera los perfiles .finance, .design, .payments, .security, .ops y .relations.

Router Onboarding flow Perfiles
Fase 4

Práctica con 3 misiones reales

60–90 min

Cada participante publica tres misiones con los héroes, revisa entregables en Sheets/Docs y ejecuta el demo del Tesorero para validar que todo funciona.

3 misiones Eval loop demo_tesorero.py
Instalación

Empieza en minutos.

El instalador configura entorno Python, dependencias, el comando global hermes y los seis perfiles. También puedes correr Partenon dentro de NVIDIA NemoClaw y habilitar las Stripe Skills oficiales.

  1. 1. Clona el repositorio.
  2. 2. Corre el instalador local o el onboard de NemoClaw.
  3. 3. Instala las Stripe Skills oficiales.
  4. 4. Copia el Google Sheet base.
  5. 5. Configura variables de entorno.
  6. 6. Define el tipo de empresa.
