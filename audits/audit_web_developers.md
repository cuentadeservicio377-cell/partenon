## Executive Summary

The `web/developers.html` page accurately describes the **vision** and repo layout, but it repeatedly presents stubs, dry-run wrappers, and conceptual APIs as already-working developer surfaces. The biggest trust risks are:

1. **Fabricated CLI and install paths.** The `hermes` command examples and `npx create-hermes@latest` installer do not exist in the repo, yet they are shown as the primary developer onboarding path.
2. **Non-existent MCP servers.** Social APIs, CRM, and NVIDIA MCP servers are advertised with brand names (Instagram, HubSpot, Salesforce, NVIDIA), but the codebase contains only dry-run stubs or no implementation at all.
3. **Wrong API surface.** The documented `/api/v1/mcp/tools` and `/api/v1/mcp/call` endpoints are not implemented; the real integration API is under `/api/v1/integrations`.
4. **Overstated credentials requirements.** `NVIDIA_API_KEY` and Google OAuth are labeled required in the env-var table even though no live NVIDIA integration exists and Google service-account auth is the actual supported path.

**Recommendation:** Add clear "vision / roadmap" labels to conceptual sections, replace the fictional CLI examples with the real `./install.sh` flow, and update the API/MCP diagrams to reflect only the implemented servers and tool names.
